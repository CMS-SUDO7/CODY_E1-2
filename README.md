주제: 나라별 수도 맞추기

이유:
- 누구나 기본적으로 알고 있는 상식
- 난이도 조절이 쉬움
- 확장성이 좋음 (국가 추가 가능)

## 클래스 설계
```bash
Quiz 클래스의 역할

Quiz 클래스는 퀴즈 1개를 표현하는 클래스이다.
하나의 퀴즈가 가져야 하는 데이터와 동작을 함께 관리한다.

포함하는 내용:

문제(question)
보기(choices)
정답(answer)
문제 출력 기능
정답 확인 기능
JSON 저장을 위한 딕셔너리 변환 기능

즉, Quiz 클래스는 개별 문제 데이터와 문제 1개의 동작을 담당한다.

QuizGame 클래스의 역할

QuizGame 클래스는 게임 전체를 관리하는 클래스이다.

포함하는 내용:

메뉴 출력
퀴즈 진행
퀴즈 추가
퀴즈 목록 출력
점수 확인
파일 저장/불러오기
예외 처리 및 안전 종료

즉, QuizGame 클래스는 여러 개의 Quiz 객체를 관리하면서 프로그램 전체 흐름을 담당한다.

책임 분리 이유

문제 1개를 다루는 역할과, 게임 전체를 운영하는 역할은 서로 다르다.
이를 하나의 클래스에 모두 넣으면 코드가 길어지고 수정이 어려워진다.

그래서 다음과 같이 책임을 분리하였다.

Quiz → 문제 1개 관리
QuizGame → 전체 게임 진행 관리

이렇게 분리하면 코드의 구조가 더 명확해지고, 기능 수정과 확장이 쉬워진다.
```
## 입력 처리 / 게임 진행 / 데이터 저장 로직 분리 기준
```
이 프로그램은 기능을 크게 3가지로 나누어 설계하였다.

1) 입력 처리

사용자로부터 값을 입력받고, 입력값이 올바른지 검사하는 부분이다.

예:

메뉴 번호 입력
정답 번호 입력
퀴즈 추가 시 문제/선택지 입력

입력 처리 로직을 별도 메서드로 분리한 이유는 다음과 같다.

중복 코드를 줄일 수 있다.
모든 입력에 대해 같은 기준으로 예외 처리를 할 수 있다.
빈 입력, 숫자 변환 실패, 범위 오류를 일관되게 처리할 수 있다.
2) 게임 진행

실제로 퀴즈를 출제하고, 사용자의 답을 확인하고, 점수를 계산하는 부분이다.

예:

문제 출력
정답/오답 판정
최종 점수 계산
최고 점수 갱신

이 부분은 프로그램의 핵심 기능이므로 입력 처리나 파일 저장과 분리하여 관리하였다.

3) 데이터 저장

퀴즈 목록과 최고 점수를 파일에 저장하고, 다시 불러오는 부분이다.

예:

state.json 저장
state.json 불러오기
파일이 없거나 손상되었을 때 기본 데이터 복구

이 부분을 따로 분리한 이유는 게임의 진행 로직과 파일 입출력 로직의 역할이 다르기 때문이다.
```
## 클래스 사용 이유와 함수와의 차이점
```
이 프로그램은 함수만으로도 만들 수 있지만, 클래스를 사용하는 것이 더 적절하다고 판단했다.

함수만 사용할 경우

함수만 사용하면 퀴즈 목록, 현재 점수, 최고 점수 같은 상태 데이터를
여러 함수 사이에서 계속 전달해야 한다.

이 경우:

전역 변수 사용이 많아질 수 있고
함수 간 의존성이 커질 수 있으며
코드 구조가 복잡해질 수 있다
클래스를 사용할 경우

클래스를 사용하면 관련된 데이터와 동작을 하나의 객체 안에 묶을 수 있다.

예:

Quiz 객체 하나가 문제 1개를 표현한다.
QuizGame 객체 하나가 전체 게임 상태를 관리한다.

이 방식의 장점은 다음과 같다.

코드 구조가 더 명확하다.
유지보수가 쉽다.
기능 추가와 확장에 유리하다.
상태 데이터를 체계적으로 관리할 수 있다.

따라서 본 프로젝트에서는 데이터와 기능을 함께 묶어 관리하기 위해 클래스 기반 구조를 사용하였다.
```
## JSON 저장 이유와 특징
```
이 프로젝트는 데이터를 state.json 파일에 저장한다.

JSON을 선택한 이유
사람이 읽고 이해하기 쉽다.
파이썬에서 다루기 쉽다.
딕셔너리와 리스트 구조를 그대로 표현할 수 있다.
별도의 데이터베이스 없이 간단하게 저장할 수 있다.
작은 규모의 프로젝트에 적합하다.

JSON의 특징
텍스트 파일 형식이다.
구조화된 데이터를 저장하기 쉽다.
여러 언어에서 공통으로 사용할 수 있다.
문제 목록 같은 리스트 데이터를 표현하기에 편리하다.

이 프로젝트에서는 퀴즈 목록과 최고 점수처럼
구조가 비교적 단순한 데이터를 저장하므로 JSON이 적합하다.
```
## 파일 입출력 시 try/except가 필요한 이유
```
파일을 읽거나 저장하는 과정에서는 여러 가지 예외가 발생할 수 있다.
따라서 try/except를 사용하여 프로그램이 비정상 종료되지 않도록 하였다.

필요한 이유
파일이 존재하지 않을 수 있다.
JSON 형식이 손상되었을 수 있다.
읽기/쓰기 권한 문제가 발생할 수 있다.
저장 중 예기치 못한 오류가 발생할 수 있다.
실패 케이스 예시
state.json이 처음부터 없는 경우
사용자가 파일 내용을 직접 수정하다가 JSON 형식을 깨뜨린 경우
잘못된 인코딩으로 인해 읽기 실패가 발생한 경우
파일 저장 중 예외가 발생한 경우

이러한 상황에서도 프로그램이 강제 종료되지 않도록 하고,

기본 퀴즈를 불러오거나
안내 메시지를 출력하거나
가능한 범위에서 복구하도록 설계하였다.
```
## state.json 데이터 구조 설계 이유
```
state.json은 프로그램 실행에 필요한 최소 상태만 저장하도록 설계하였다.

이렇게 설계한 이유

리스트 안에 객체 형태로 저장하여 확장하기 쉽다.
새로운 퀴즈를 추가해도 구조를 유지할 수 있다.

즉, 필요한 정보만 일관된 구조로 저장하는 것을 목표로 하였다.
```
## JSON 파싱 실패 시 복구/대응 방안
```
JSON 파싱 실패는 보통 state.json 파일이 손상되었거나 형식이 잘못되었을 때 발생한다.

대응 방법

필요하면 새 state.json 파일로 다시 저장하여 복구한다.
이렇게 처리한 이유

따라서 강제 종료보다는 기본 데이터로 복구하여 계속 사용할 수 있도록 하는 방식이 더 적절하다.

## 브랜치 분리 이유와 병합의 의미
```
Git에서 브랜치를 분리한 이유는
기존 메인 코드에 영향을 주지 않고 새로운 기능을 안전하게 개발하기 위해서이다.

예를 들어 퀴즈 풀기 기능을 개발할 때 main 브랜치에서 바로 작업하면
개발 도중 프로그램이 불완전한 상태가 될 수 있다.
그래서 별도의 브랜치에서 기능을 구현하고 테스트한 뒤, 완료되면 main에 병합하였다.

브랜치 분리의 장점
안정적인 메인 코드 유지
기능 단위 작업 가능
실수했을 때 영향 범위 축소
협업 시 충돌 감소
병합의 의미

병합은 별도 브랜치에서 개발한 기능을 main 브랜치에 반영하는 과정이다.
즉, 분리해서 개발한 기능을 최종 코드에 통합하는 작업이다.
```
## clone과 pull 실습 설명
```
이번 프로젝트에서는 Git 명령어 사용 경험을 위해 clone과 pull도 수행하였다.

clone

원격 저장소를 다른 로컬 폴더에 복제하는 명령어이다.
이를 통해 같은 프로젝트를 다른 작업 공간에서 다시 내려받아 확인할 수 있다.

pull

원격 저장소에 반영된 최신 변경사항을 현재 로컬 저장소로 가져오는 명령어이다.
예를 들어, clone한 폴더에서 README를 수정하고 push한 뒤
기존 작업 폴더에서 git pull을 하면 해당 변경사항을 받아올 수 있다.

이 과정을 통해 원격 저장소와 로컬 저장소의 동기화 흐름을 확인할 수 있다.
```
## 요구사항 변경 시 수정해야 할 파일/클래스
```
프로그램의 구조를 파일과 클래스 단위로 나누었기 때문에
요구사항이 바뀌었을 때 어느 부분을 수정해야 하는지 비교적 명확하다.

예시 1) 객관식 4지선다 → 5지선다 변경

수정 대상:

Quiz 클래스의 선택지 구조
퀴즈 출력 방식
정답 입력 범위 검사
퀴즈 추가 입력 부분

예시 2) 저장 형식 변경

JSON 대신 다른 저장 방식을 사용하려면 다음을 수정해야 한다.

수정 대상:

quiz_game.py의 저장/불러오기 메서드
README.md의 데이터 파일 설명

예시 3) 점수 계산 방식 변경

예를 들어 정답 수가 아니라 백분율 점수를 저장하려면 다음을 수정해야 한다.

수정 대상:

점수 계산 로직
최고 점수 비교 로직
저장 데이터 구조
README 설명
